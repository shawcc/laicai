# 动态发版与免费云架构方案

## 1. 目标
在网站运行过程中，可以安全更新内容和功能，不影响线上用户参与竞猜、锁票、结算和排行榜展示。

推荐免费或低成本组合：
- **GitHub**：代码仓库、Issue、PR、分支保护、GitHub Actions。
- **Vercel**：前端部署、Preview Deployment、Production Deployment、环境变量。
- **Supabase**：PostgreSQL、Auth 用户系统、Storage、Edge Functions、Row Level Security。

## 2. 环境划分
| 环境 | 分支 | Vercel | Supabase | 用途 |
|------|------|--------|----------|------|
| 本地 | 任意 | localhost | 本地或 dev project | 开发调试 |
| 预览 | Pull Request | Preview URL | Supabase dev | 验证 UI、接口、数据迁移 |
| 生产 | main | Production URL | Supabase prod | 真实用户访问 |

关键原则：
- 所有功能先进入 PR，Vercel 自动生成预览地址。
- 预览环境验证通过后再合并到 `main`。
- 生产数据库和预览数据库分离，避免测试污染线上。

## 3. 动态发版策略

### 3.1 内容动态更新
适合题目、新闻、公共 AI 情报、公告、活动规则。
- 放入 Supabase 数据表，由运营后台直接更新。
- 前端通过 Supabase API 或业务 API 读取。
- 不需要重新发版。
- 可以加 `published_at`、`status`、`version` 控制发布状态。

### 3.2 功能动态更新
适合新页面、新玩法、新结算规则。
- 使用 Feature Flag 控制功能开关。
- 新代码先部署，但默认不对所有用户开放。
- 通过 Supabase `feature_flags` 表控制灰度比例、用户白名单或管理员开关。
- 出问题时关闭开关，无需回滚代码。

### 3.3 不中断发版
Vercel 每次部署都是不可变版本：
- 新版本构建完成后才切换流量。
- 构建失败不会影响当前线上版本。
- 如线上异常，可在 Vercel 一键回滚到旧 Deployment。

## 4. 数据库迁移策略
Supabase 使用 PostgreSQL，迁移必须遵循“向前兼容”：
1. 新增字段时先允许 `NULL` 或设置默认值。
2. 先发兼容旧字段和新字段的代码。
3. 数据回填完成后，再逐步移除旧字段。
4. 结算、锁票、积分相关表禁止直接破坏性修改。

建议核心表：
```sql
profiles
questions
question_options
predictions
ai_players
evidence_packages
evidence_sources
score_events
leaderboard_snapshots
feature_flags
share_cards
audit_logs
```

## 5. 用户系统
使用 Supabase Auth：
- 支持邮箱 Magic Link、Google、GitHub、Apple 等 OAuth。
- 登录后创建 `profiles` 资料表。
- 用户可以设置昵称、头像、国家/地区、支持球队。
- 使用 Row Level Security 保护用户资料和预测记录。

基础 RLS 规则：
- 用户只能更新自己的 `profiles`。
- 用户只能在锁票前创建或修改自己的 `predictions`。
- 所有人可以读取已发布题目、AI 预测、排行榜和公开情报。
- 管理员角色才能创建题目、开奖、结算和作废题目。

## 6. 社交传播设计

### 6.1 可传播场景
- 猜中啦：用户答对题目后生成战报卡。
- 击败 AI：用户积分超过某个 AI 选手。
- 登榜：用户进入日榜、周榜或总榜前 100。
- 连续命中：用户达成 3 连中、5 连中、10 连中。
- 冷门命中：用户答对低票数选项。

### 6.2 分享卡内容
分享卡建议生成 1080x1350 竖图，适合朋友圈、小红书、Instagram。
- 用户昵称和头像。
- 命中题目和答案。
- 本题得分、总积分、排名变化。
- 击败的 AI 选手。
- 二维码或短链接。
- 活动品牌标识和赛事主题视觉。

### 6.3 实现方式
免费优先方案：
- 前端用 `html-to-image` 把分享卡 DOM 转 PNG。
- Supabase Storage 保存生成图片。
- `share_cards` 表保存图片 URL、用户、题目和分享文案。

更稳定方案：
- 使用 Vercel Edge Function 或 Supabase Edge Function 生成 Open Graph Image。
- 分享链接带 `shareId`，社交平台抓取 OG 图片。

## 7. GitHub 工作流
建议分支：
- `main`：生产分支。
- `develop`：集成分支，可选。
- `feature/*`：功能开发。
- `hotfix/*`：紧急修复。

PR 检查：
```yaml
name: Check
on: [pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run check
      - run: npm run build
```

## 8. Vercel 配置
环境变量：
```bash
VITE_SUPABASE_URL=
VITE_SUPABASE_ANON_KEY=
AI_GATEWAY_URL=
```

注意：
- `VITE_SUPABASE_ANON_KEY` 可以放前端，但必须配合 RLS。
- 大模型 API Key 不能放前端，只能放服务端或 Edge Function。
- 生产环境和预览环境使用不同 Supabase Project。

## 9. Supabase Edge Functions
建议后端轻服务：
- `refresh-evidence`：刷新公共 AI 情报。
- `submit-ai-prediction`：触发 AI 选手答题。
- `settle-question`：开奖结算。
- `generate-share-card`：生成分享图或 OG 图。

## 10. 灰度与回滚
灰度方式：
- 按用户 ID hash 灰度 5%、20%、50%、100%。
- 管理员和内部用户白名单。
- 按功能开关逐个开放。

回滚方式：
- UI 问题：Vercel 回滚上一个 Deployment。
- 功能问题：关闭 Supabase feature flag。
- 数据问题：执行补偿脚本，禁止直接删积分事件。

## 11. 上线顺序
1. 建 GitHub 仓库，接入 Vercel。
2. 创建 Supabase dev/prod 项目。
3. 接入 Supabase Auth 和 profiles。
4. 把题目、预测、积分事件从 Mock 数据迁移到 Supabase。
5. 接入 Feature Flag。
6. 接入公共 AI 情报员和 AI 选手 Edge Functions。
7. 接入分享卡生成和社交 OG 图片。
8. 建立 PR 预览、灰度、回滚和审计流程。
